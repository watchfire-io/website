This page is a snapshot of intent, not a commitment. Watchfire is open
source and small, and priorities shift with what users actually run into.
Everything below is grounded in the [Phoenix (v6.0.0)](/docs/changelog)
release on top of the [Beacon (v4.0.0)](/docs/changelog#400-beacon)
foundation — the [notification bus](/docs/glossary#notification-bus), the
[outbound relay framework](/docs/concepts/integrations#outbound), the
[inbound HTTP server](/docs/concepts/integrations#inbound), the
[insights pipeline](/docs/concepts/insights), and the dynamic system
tray. If something here matters to you, jump to
[How to influence the roadmap](#how-to-influence-the-roadmap).

<Callout title="Roadmap, not commitment">
Items are listed by intent, not by date. We don't promise quarters and
won't ship anything that pretends to be a feature when it isn't. If a
"Next" item is blocking you, the fastest path is an issue or a PR — see
the bottom of the page.
</Callout>

## Now (shipped — v6.0.0 Phoenix)

Phoenix lands the `project.yaml` data-loss fix, the flock-based
singleton-daemon hardening, and Cursor Agent CLI as a sixth first-class
backend, plus a TUI rewrite that touches Project Settings, the agent
pane, and a half-dozen new overlays. The
[full changelog](/docs/changelog) has every entry; the short version:

- **Cursor Agent CLI as a sixth backend** — `cursor-agent` joins
  Claude Code, Codex, opencode, Gemini, and Copilot through the same
  `Backend` interface, with per-session home, symlinked auth, and
  JSONL transcripts. See [Supported Agents](/docs/concepts/supported-agents#cursor-agent).
- **Atomic YAML writes** — `config.SaveYAML` now writes to a sibling
  tmp file, `fsync`s, then `os.Rename`s into place. Closes the
  `project.yaml` data-loss race that let a partially-loaded struct
  round-trip back to disk and wipe project metadata.
- **Singleton daemon via flock** — `~/.watchfire/daemon.lock` is held
  for the daemon's full lifetime, so two `watchfired` processes can
  no longer race onto separate dynamic ports and spawn duplicate
  menu-bar tray icons.
- **TUI Project Settings sidebar refactor** — the per-project
  Settings tab gets a macOS-style sidebar + content-pane layout with
  seven sections (General, Automation, Notifications, Integrations,
  Metadata, Secrets, Danger zone) and a `/` search overlay that jumps
  straight to a matching control.
- **Project-scoped notifications + integrations** — per-event,
  quiet-hours, sound, Slack channel, Discord guild ID, and GitHub
  auto-PR membership all override globals on a per-project basis from
  inside the new sidebar.
- **TUI Trash filter mode** — `D` flips the Tasks tab between active
  and soft-deleted subsets, with `u` restoring and `x` permanently
  deleting (refused if the `watchfire/<n>` branch reports unmerged
  work).
- **TUI Definition tab `$EDITOR` shellout** — `e` opens the project
  definition in your editor of choice via `tea.ExecProcess`, diffs
  the result, and dispatches `UpdateProject` only when content
  changed.
- **TUI Branches overlay (`Ctrl+B`)** — list every `watchfire/<n>`
  branch with merge/worktree status, age, and orphan detection. `m`
  merges, `x` deletes, `X` force-deletes, `P` prunes all
  merged-orphans.
- **TUI text-select mode (`Ctrl+T`)** — toggle Bubble Tea mouse
  capture so host-native click-and-drag selection works inside the
  TUI without leaving the session.
- **TUI agent pane terminal-emulator swap** — the agent pane now
  runs `charmbracelet/x/vt` against `SubscribeRawOutput`, restoring
  full xterm-compat alt-screen + scroll regions and fixing the
  long-standing "input lands at top" tear bug under claude code.
  Up to 5000 lines of real scrollback.

## Next (v7.0)

Plausible next steps that fall naturally out of the Phoenix architecture
— written as direction, not as scoped work.

- **Richer report templates** — the Markdown export already runs through
  templates under `internal/daemon/insights/templates/`. Expanding the
  template set (per-agent breakdowns, cost-per-task heatmaps, custom
  date ranges) is incremental work on the existing pipeline.
- **Additional outbound adapters** — the
  [outbound `Adapter` interface](/docs/concepts/integrations#outbound)
  is intentionally generic. New adapters (e.g. Microsoft Teams, Linear,
  generic SMTP) plug into the same `Dispatcher`, retry timing, and
  circuit breaker without touching the bus.
- **More inbound provider handlers** — the
  [inbound HTTP server](/docs/concepts/integrations#inbound) is built
  around `RegisterProvider(method, path, handler)`. New verifiers and
  dispatch handlers (more GitHub event types, Slack event subscriptions
  beyond slash commands, generic webhook ingest) are additive.
- **Expanded backend matrix** — the
  [pluggable agent backend interface](/docs/concepts/supported-agents)
  already covers Claude Code, OpenAI Codex, opencode, Gemini CLI,
  GitHub Copilot CLI, and Cursor. Filling in the Copilot and Cursor
  metrics stubs (so token and cost rollups stop being partial) and
  adding new backends as they appear are natural follow-ups.
- **GUI parity for the new TUI overlays** — the Project Settings
  sidebar, Branches overlay, and Trash filter mode all landed in the
  TUI first. Bringing them across to the Electron client is mostly
  layout work on top of the existing gRPC surface.
- **Concurrent-edit handling for the Definition `$EDITOR` flow** —
  Phoenix's last-write-wins is fine for v6.x but a preconditioned
  `UpdateProject` (with an `UpdatedAt` field on the proto) would let
  two simultaneous editors converge instead of clobber.

## Later (exploratory)

These are honest "we're thinking about it" items — open questions, not
queued work. They exist on this list because users keep asking; they
are not scoped, sized, or scheduled.

- **Cloud / hosted daemon mode** — a hosted `watchfired` so teams can
  drive runs from any machine, not just the one the daemon was
  installed on. The daemon is already a network service; the hard parts
  are auth, multi-tenancy, and what "sandbox" means off your laptop.
- **Multi-user fleet view** — shared dashboards, shared insights,
  shared integration credentials across a small team. Today the daemon
  is
  single-user by design; opening that up means redesigning identity and
  the secret store, not just adding a column to a table.
- **Deeper editor integrations** — beyond the bundled VS Code shells
  inside the GUI, there is interest in tighter integration with editors
  (cursor positioning, in-editor task creation, diff overlays). Whether
  this lives in Watchfire or in editor extensions that talk to the
  daemon over gRPC is still an open call.

## How to influence the roadmap

Roadmaps move when users push them. The fastest paths:

- **Feature requests** —
  [open an issue](https://github.com/watchfire-io/watchfire/issues) on
  the Watchfire repo. The
  [bug report and feature templates](https://github.com/watchfire-io/watchfire/issues/new/choose)
  cover the common shapes; for anything that doesn't fit, an empty
  issue with a clear title is fine.
- **Open-ended discussion** — start a thread in
  [GitHub Discussions](https://github.com/watchfire-io/watchfire/discussions)
  if the idea isn't yet shaped enough to be an issue. Discussions are
  also where the "Later" items above tend to get refined.
- **Release history** — the [changelog](/docs/changelog) is the
  authoritative record of what has actually shipped. If you're trying
  to figure out whether a roadmap item has graduated to the changelog,
  that's the page to check.
- **Contribute directly** — see the
  [contributing guide](/docs/contributing) for how to land a PR. The
  `good first issue` and `help wanted` labels are the easiest entry
  points.
