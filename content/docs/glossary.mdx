## Release codenames

### Ember

The 0.x and 1.0 release line that shipped the original Claude Code-only daemon, CLI/TUI, and GUI on macOS, Linux, and Windows. [Changelog ‚Üí](/docs/changelog)

### Spark

The 2.x release line that introduced the pluggable [agent backend](#backend) interface and shipped Claude Code, OpenAI Codex, opencode, and Gemini CLI as first-class backends. [Changelog ‚Üí](/docs/changelog)

### Blaze

The 3.x release line that added GitHub Copilot CLI as a fifth first-class backend. [Changelog ‚Üí](/docs/changelog)

### Beacon

The 4.x release line ‚Äî the consolidated dashboard, notifications, insights, and integrations release. [Changelog ‚Üí](/docs/changelog)

## Agent modes

### Chat mode

Free-form interactive session with the coding agent at project root, with no task context. [Agent modes ‚Üí](/docs/concepts/agent-modes#chat)

### Task mode

Single-task execution where the agent receives a task's prompt and acceptance criteria and works inside an isolated [worktree](#worktree). [Agent modes ‚Üí](/docs/concepts/agent-modes#task)

### Start All mode

Sequential execution of every `ready` task in turn, each in its own worktree. [Agent modes ‚Üí](/docs/concepts/agent-modes#start-all)

### Wildfire mode

Fully autonomous three-phase loop ‚Äî execute ready tasks, refine drafts, generate new tasks, then repeat until no work remains. [Agent modes ‚Üí](/docs/concepts/agent-modes#wildfire)

### Generate Definition mode

One-shot mode where the agent analyses the codebase and writes the project definition. [Agent modes ‚Üí](/docs/concepts/agent-modes#generate-definition)

### Generate Tasks mode

One-shot mode where the agent reads the project definition and creates a batch of task files. [Agent modes ‚Üí](/docs/concepts/agent-modes#generate-tasks)

## Components

### Daemon

The always-on Go process (`watchfired`) that owns orchestration, PTY management, git worktree workflows, the gRPC server, and the system tray. [Daemon doc ‚Üí](/docs/components/daemon)

### CLI/TUI

The `watchfire` binary ‚Äî project-scoped CLI commands plus a Bubbletea-based interactive split-view TUI. [CLI/TUI doc ‚Üí](/docs/components/cli)

### GUI

The Electron desktop client (`Watchfire.app`) that shows every project across the fleet at once. [GUI doc ‚Üí](/docs/components/gui)

### Echo server

The inbound HTTP server inside the daemon (`internal/daemon/echo`) that receives signed requests from external services like Slack, Discord, and GitHub. [Integrations doc ‚Üí](/docs/concepts/integrations#inbound)

### Relay dispatcher

The outbound delivery component (`internal/daemon/relay`) that subscribes to the [notification bus](#notification-bus) and fans events out to every registered [adapter](#adapter). [Integrations doc ‚Üí](/docs/concepts/integrations#outbound)

### System tray

The dynamic menu-bar icon rendered by `internal/daemon/tray` that surfaces per-project status, today's notifications, and quick stop/quit actions. [Daemon doc ‚Üí](/docs/components/daemon)

## Project model

### Project

A registered Watchfire workspace ‚Äî one git repository plus its `.watchfire/` directory and entry in `~/.watchfire/projects.yaml`. [Projects and Tasks ‚Üí](/docs/concepts/projects-and-tasks)

### Task

A unit of work described by a YAML file in `.watchfire/tasks/`, flowing through `draft ‚Üí ready ‚Üí done` and optionally pinned to a specific [backend](#backend). [Projects and Tasks ‚Üí](/docs/concepts/projects-and-tasks#task-schema)

### Project YAML

The `project.yaml` file inside `.watchfire/` that holds project id, default agent, sandbox mode, auto-merge / auto-delete-branch / auto-start flags, and the project definition. [Projects and Tasks ‚Üí](/docs/concepts/projects-and-tasks#project-configuration)

### Task YAML

A single `<n>.yaml` file inside `.watchfire/tasks/` that carries `task_id`, `task_number`, `title`, `prompt`, `acceptance_criteria`, `status`, `success`, and an optional per-task `agent` override. [Projects and Tasks ‚Üí](/docs/concepts/projects-and-tasks#task-schema)

### .watchfire directory

The per-project gitignored directory that holds `project.yaml`, `tasks/`, `memory.md`, `secrets/`, and active `worktrees/`. [Projects and Tasks ‚Üí](/docs/concepts/projects-and-tasks#directory-structure)

## Isolation

### Worktree

A per-task git worktree at `.watchfire/worktrees/<task_number>/` on branch `watchfire/<task_number>`, where the agent makes changes without touching the main working tree. [Worktrees doc ‚Üí](/docs/concepts/worktrees)

### Sandbox

The platform-specific isolation layer Watchfire wraps every agent process in, configured via `sandbox: auto | seatbelt | landlock | bwrap | none`. [Sandboxing doc ‚Üí](/docs/concepts/sandboxing)

### Seatbelt

The macOS sandbox backend (`sandbox-exec` / Apple's App Sandbox) that enforces a kernel-level filesystem and credential-directory deny-list generated from the runtime `SandboxPolicy`. [Sandboxing doc ‚Üí](/docs/concepts/sandboxing#macos--seatbelt)

### Landlock

The preferred Linux sandbox backend, using the kernel's Landlock LSM (5.13+) with no external dependencies. [Sandboxing doc ‚Üí](/docs/concepts/sandboxing#linux--landlock)

### Bubblewrap

The fallback Linux sandbox backend (`bwrap`) that uses mount namespaces with a read-only root and a writable project directory. [Sandboxing doc ‚Üí](/docs/concepts/sandboxing#linux--bubblewrap-fallback)

### Backend

A single Go implementation of the `Backend` interface in `internal/daemon/agent/backend/` ‚Äî one per supported coding agent (`claude-code`, `codex`, `opencode`, `gemini`, `copilot`) ‚Äî that owns binary resolution, command construction, system-prompt delivery, transcript discovery, and sandbox extras. [Supported agents ‚Üí](/docs/concepts/supported-agents)

## Beacon vocabulary

### Notification bus

The typed channel-fan-out `Bus` in `internal/daemon/notify` that emits stable-ID notification events to every subscriber, with a JSONL fallback at `~/.watchfire/logs/<project_id>/notifications.log`. [Daemon doc ‚Üí](/docs/components/daemon)

### Notification kind

The discriminator on every notification event ‚Äî `TASK_FAILED`, `RUN_COMPLETE`, or `WEEKLY_DIGEST` ‚Äî that determines toast routing, channel rendering, and gating. [Integrations doc ‚Üí](/docs/concepts/integrations#outbound)

### Notification preferences

The `defaults.notifications` block in `~/.watchfire/settings.yaml` controlling master toggle, per-event toggles, sounds, volume, quiet hours, and per-project mute. [Daemon doc ‚Üí](/docs/components/daemon)

### Idempotency cache

The 1000-entry / 24-hour LRU+TTL cache in `internal/daemon/echo/idempotency.go` that drops duplicate inbound deliveries via `Seen(key)`. [Integrations doc ‚Üí](/docs/concepts/integrations#idempotency-cache)

### Signature verification

The constant-time HMAC-SHA256 (GitHub, Slack) and Ed25519 (Discord) verifiers in `internal/daemon/echo/verify.go` that authenticate inbound requests with a 5-minute drift window. [Integrations doc ‚Üí](/docs/concepts/integrations#signature-verification)

### Insights

The per-project and cross-project aggregation surface that turns `<n>.metrics.yaml` files into KPIs, agent donuts, duration histograms, and rollup cards. [Insights doc ‚Üí](/docs/concepts/insights)

### Metrics

The `<n>.metrics.yaml` siblings in `.watchfire/tasks/` capturing duration, exit reason, agent, tokens, and cost for every completed task. [Insights doc ‚Üí](/docs/concepts/insights#per-task-metrics-capture)

### Diff viewer

The structured `FileDiffSet` produced by `internal/daemon/diff` (capped at 10 000 lines) that resolves pre-merge diffs against `<merge-base>...HEAD` and post-merge diffs against the merge commit, surfaced in the GUI Inspect tab and the TUI overlay bound to <kbd>d</kbd>. [Worktrees doc ‚Üí](/docs/concepts/worktrees#merge-behavior)

### Focus events

The `DaemonService.SubscribeFocusEvents` stream that routes click-throughs from the system tray and OS notifications to the right project, task, or digest in a connected client.

### Status bar

The muted dashboard line `N working ¬∑ N needs attention ¬∑ N idle ¬∑ N done today` rendered above the project grid in the GUI. [GUI doc ‚Üí](/docs/components/gui)

### Filter chips

The dashboard pill chips (`All`, `Working`, `Needs attention`, `Idle`, `Has ready tasks`) whose selection persists in `localStorage[wf-dashboard-filter]`. [GUI doc ‚Üí](/docs/components/gui)

### Layout toggle

The dashboard `LayoutGrid` / `Rows3` toggle that switches the GUI dashboard between project cards and a one-row-per-project list, persisted in `localStorage[wf-dashboard-layout]`. [GUI doc ‚Üí](/docs/components/gui)

## Integrations

### Adapter

A single implementation of the relay `Adapter` interface ‚Äî one per outbound destination (webhook, Slack, Discord, GitHub auto-PR) ‚Äî owning its template rendering, retry timing, and circuit breaker. [Integrations doc ‚Üí](/docs/concepts/integrations#outbound)

### Webhook adapter

The generic adapter that POSTs the canonical `notify.Notification` payload to a user-supplied URL with an `X-Watchfire-Signature: sha256=<hex>` HMAC over the raw body. [Integrations doc ‚Üí](/docs/concepts/integrations#webhook-adapter)

### Slack Block Kit

The three header / section / context / actions envelopes (`TASK_FAILED`, `RUN_COMPLETE`, `WEEKLY_DIGEST`) rendered by `internal/daemon/relay/slack.go` for the Slack adapter. [Integrations doc ‚Üí](/docs/concepts/integrations#slack-adapter)

### Discord rich embed

The three project-color-tinted embed envelopes rendered by `internal/daemon/relay/discord.go` for the Discord adapter, with a defensive 4000-rune description trim. [Integrations doc ‚Üí](/docs/concepts/integrations#discord-adapter)

### GitHub auto-PR

The opt-in per-project end-of-task lifecycle in `internal/daemon/git/pr.go::OpenPR` that pushes the `watchfire/<n>` branch and opens a pull request via `gh api`. [Integrations doc ‚Üí](/docs/concepts/integrations#github-auto-pr)

### Slash command router

The transport-agnostic `Route(ctx, cmd, subcmd, rest, CommandContext) CommandResponse` in `internal/daemon/echo/commands.go` that powers `status`, `retry <task>`, and `cancel <task>` across every inbound transport. [Integrations doc ‚Üí](/docs/concepts/integrations#command-router)
